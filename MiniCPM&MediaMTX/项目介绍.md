# 基于 Atlas 200 与 MiniCPM-V 的嵌入式多模态视觉分析系统设计与实现

---

## 1. 设计题目简介

本设计实现一个基于嵌入式边缘计算设备、PC 端多模态大模型和嵌入式交互终端的智能视觉分析系统。系统整体由 **Atlas 200 AI 开发板、PC 主机、IMX6 嵌入式实验箱** 三部分组成。

其中，Atlas 200 负责接入视频源并进行目标检测与视频推流；PC 端运行 MediaMTX 流媒体服务和 MiniCPM-V 4.0 多模态大模型，负责接收视频流、截取当前画面并结合用户输入的文字提示进行图像语义分析；IMX6 嵌入式实验箱运行 Qt 交互程序，作为用户操作终端，负责输入分析指令、发送分析请求并显示分析结果。

本系统的主要目标是完成一个“边缘视觉感知 + PC 大模型分析 + 嵌入式人机交互”的完整计算系统。用户可以在 IMX6 的 Qt 界面中输入文字提示，例如“请分析画面中是否存在车辆”“判断当前画面中是否有行人”等，然后点击“分析”按钮。PC 端收到请求后，从 MediaMTX 视频流中获取当前帧图像，调用 MiniCPM-V 4.0 对图像和用户提示进行综合分析，最后将分析结果返回给 IMX6 并在界面中显示。

本设计综合涉及嵌入式开发、网络通信、视频流传输、AI 推理、多模态图像理解和 Qt 图形界面设计，能够较好体现计算系统软硬件协同设计与实现过程。

## 2. 整体设计及说明

### 2.1 系统总体架构

系统总体采用三端协同的分布式架构，分别为：

1. **Atlas 200 端**
   负责视频采集、目标检测、画面处理和视频推流。

2. **PC 端**
   负责运行 MediaMTX 流媒体服务器、MiniCPM-V 4.0 多模态大模型以及后端分析服务。

3. **IMX6 端**
   负责运行 Qt 图形界面，完成用户输入、指令发送和结果显示。

系统整体数据流如下：

```text
摄像头 / 视频文件
      ↓
Atlas 200
  视频采集 + YOLO26 目标检测 + 画面绘制
      ↓ RTSP 推流
PC 端 MediaMTX
      ↓
PC 后端分析服务
  抓取当前帧 + 调用 MiniCPM-V 4.0
      ↓ HTTP 返回 JSON
IMX6 Qt 程序
  显示分析结果
```

整体系统结构图可以描述为：

```text
                 ┌────────────────────┐
                 │      交换机/路由器    │
                 └───────┬───────┬────┘
                         │       │
          ┌──────────────┘       └──────────────┐
          │                                      │
 ┌────────────────┐                    ┌────────────────┐
 │   Atlas 200     │                    │      IMX6       │
 │  视频采集与推流  │                    │   Qt交互界面     │
 └───────┬────────┘                    └───────┬────────┘
         │ RTSP推流                             │ HTTP请求
         │                                      │
         └──────────────┐       ┌──────────────┘
                        ↓       ↓
                  ┌────────────────────┐
                  │        PC端         │
                  │ MediaMTX + MiniCPM  │
                  │ FastAPI分析服务     │
                  └────────────────────┘
```

由于 PC 只有一个网口，为保证 Atlas 200、PC 和 IMX6 三台设备能够稳定通信，系统采用交换机或路由器组成局域网。三台设备均接入同一个局域网，通过固定 IP 或 DHCP 分配 IP 地址实现互联。

### 2.2 硬件组成

本系统主要使用以下硬件设备：

| 设备               | 作用                               |
| ---------------- | -------------------------------- |
| Atlas 200 AI 开发板 | 负责边缘端视频处理、目标检测和视频推流              |
| PC 主机            | 运行 MediaMTX、MiniCPM-V 4.0 和后端服务  |
| RTX 4060 8GB 显卡  | 为 MiniCPM-V 4.0 多模态模型提供 GPU 推理能力 |
| IMX6 嵌入式实验箱      | 运行 Qt 交互程序，实现人机交互                |
| 交换机/路由器          | 连接 Atlas 200、PC 和 IMX6，组成局域网     |
| USB 摄像头 / 视频文件   | 作为视频输入源                          |

其中，Atlas 200 作为边缘计算节点，适合部署目标检测、视频编码和推流任务。PC 端具备更强的计算能力，尤其配备 RTX 4060 显卡，因此用于运行 MiniCPM-V 4.0 多模态大模型。IMX6 嵌入式实验箱主要承担交互功能，避免在 IMX6 上运行复杂 AI 模型，从而降低嵌入式端资源压力。

### 2.3 网络连接设计

由于系统中包含三台需要互相通信的设备，因此采用局域网方式连接。推荐使用一个普通千兆交换机或路由器，将 Atlas 200、PC 和 IMX6 分别通过网线连接到交换机。

网络规划如下：

| 设备        | IP 地址示例      | 端口/服务                     |
| --------- | ------------ | ------------------------- |
| PC        | 192.168.1.10 | MediaMTX: 8554，后端服务: 8000 |
| Atlas 200 | 192.168.1.11 | 向 PC 推送 RTSP 视频流          |
| IMX6      | 192.168.1.12 | 访问 PC 后端接口                |

Atlas 200 将处理后的视频流推送到 PC：

```text
rtsp://192.168.1.10:8554/live
```

IMX6 通过 HTTP 请求访问 PC 的分析服务：

```text
http://192.168.1.10:8000/analyze
```

这样设计后，Atlas 200 不需要直接和 IMX6 通信，IMX6 也不需要直接读取视频流。PC 作为中心节点，统一完成视频流接收、大模型分析和结果返回。

### 2.4 软件功能模块设计

系统软件部分主要分为四个模块：

1. **Atlas 200 视频处理与推流模块**
2. **PC 端 MediaMTX 视频流服务模块**
3. **PC 端 MiniCPM-V 图像分析模块**
4. **IMX6 Qt 交互模块**

#### 2.4.1 Atlas 200 视频处理与推流模块

Atlas 200 端主要完成视频输入、目标检测、图像绘制和推流功能。输入源可以是 USB 摄像头，也可以是本地视频文件。Atlas 200 端运行基于 CANN/AscendCL 的 YOLO26 目标检测程序，对每一帧图像进行推理，得到车辆、行人等目标的位置和类别。

检测完成后，程序可以在原始画面上绘制检测框、类别信息和置信度，然后将处理后的画面编码为视频流，通过 RTSP 协议推送到 PC 端 MediaMTX。

该模块主要功能包括：

```text
视频读取 → 图像预处理 → YOLO26推理 → 后处理 → 结果绘制 → 视频编码 → RTSP推流
```

#### 2.4.2 PC 端 MediaMTX 流媒体服务模块

PC 端运行 MediaMTX，用于接收 Atlas 200 推送过来的视频流。MediaMTX 作为轻量级流媒体服务器，负责视频流的接收、管理和分发。

Atlas 200 将 RTSP 流推送到：

```text
rtsp://192.168.1.10:8554/live
```

PC 后端分析程序在需要分析时，可以从该地址读取视频帧。这样可以将视频推流过程和大模型分析过程解耦，使系统结构更加清晰。

#### 2.4.3 PC 端 MiniCPM-V 图像分析模块

PC 端是系统的核心分析节点。该端运行 MiniCPM-V 4.0 int4 多模态大模型，使用 RTX 4060 8GB 显卡进行推理。

当 IMX6 发来分析请求后，PC 后端服务执行以下步骤：

```text
接收用户文字提示
      ↓
从 MediaMTX 的 RTSP 视频流中截取当前帧
      ↓
将当前帧图像和用户提示词输入 MiniCPM-V 4.0
      ↓
生成图像理解结果
      ↓
通过 HTTP 返回给 IMX6
```

例如，IMX6 发送的请求内容为：

```json
{
  "prompt": "请判断画面中是否有车辆，并说明车辆大致位置。"
}
```

PC 端返回：

```json
{
  "status": "ok",
  "result": "画面中可以看到一辆车辆，位于图像中央偏下区域，车辆前方空间较为空旷。"
}
```

为了提高回答稳定性，可以在 PC 后端为用户输入添加默认系统提示词：

```text
你是一个嵌入式视觉分析助手。请基于当前画面回答用户问题，回答要简洁、明确。
用户问题：{用户输入}
```

#### 2.4.4 IMX6 Qt 交互模块

IMX6 嵌入式实验箱运行 Qt 程序，主要负责用户交互。界面中设计输入框、分析按钮和结果显示区域。

用户可以在输入框中输入文字提示，例如：

```text
请分析当前画面中是否有行人。
```

然后点击“分析”按钮。Qt 程序通过 HTTP POST 请求将用户提示发送到 PC 端分析服务。PC 分析完成后返回 JSON 格式结果，Qt 程序解析返回内容并显示在界面中。

IMX6 端不承担复杂 AI 推理任务，只负责交互显示，这样可以降低嵌入式端计算负载，提高系统稳定性。

## 3. 自己承担部分具体设计

在本系统中，我主要承担以下部分的设计与实现工作：

1. Atlas 200 端目标检测与视频推流程序设计；
2. PC 端 MediaMTX 流媒体服务配置；
3. PC 端 MiniCPM-V 多模态分析服务设计；
4. IMX6 端 Qt 交互界面设计；
5. 三端网络通信与系统联调。

下面对本人承担的主要模块进行具体说明。

---

### 3.1 Atlas 200 端视频推流设计

Atlas 200 端负责视频输入与边缘目标检测。程序首先从 USB 摄像头或本地视频文件中读取图像帧，然后将图像缩放到 YOLO26 模型所需输入尺寸。模型推理完成后，程序解析输出结果，得到目标类别、置信度和检测框坐标。

为了便于用户观察检测结果，在推流前将检测框和类别信息直接绘制到视频帧上。随后，程序将处理后的帧送入视频编码模块，并通过 RTSP 推送到 PC 端 MediaMTX。

该模块设计重点包括：

* 保证视频读取和推理流程连续；
* 保证推流地址与 PC 端 MediaMTX 地址一致；
* 保证视频帧率尽量稳定；
* 推流画面中应包含检测结果，方便演示。

推流地址设计为：

```text
rtsp://192.168.1.10:8554/live
```

其中 `192.168.1.10` 为 PC 端 IP 地址，`live` 为视频流名称。

---

### 3.2 PC 端后端分析服务设计

PC 端除了运行 MediaMTX 外，还需要运行一个后端服务程序。该程序可以使用 Python 的 FastAPI 或 Flask 实现。后端服务主要提供 `/analyze` 接口，供 IMX6 Qt 程序调用。

接口设计如下：

```http
POST /analyze
Content-Type: application/json
```

请求内容：

```json
{
  "prompt": "请分析当前画面中是否存在车辆。"
}
```

返回内容：

```json
{
  "status": "ok",
  "result": "当前画面中存在车辆，车辆位于画面中央偏下位置。"
}
```

后端服务处理流程如下：

```text
接收 IMX6 请求
      ↓
读取 prompt 字段
      ↓
从 RTSP 视频流中抓取一帧
      ↓
调用 MiniCPM-V 4.0 进行图文分析
      ↓
返回 JSON 结果
```

为了提高系统鲁棒性，后端服务还需要处理异常情况，例如：

* 视频流读取失败；
* MiniCPM 模型加载失败；
* 用户 prompt 为空；
* 分析超时；
* 网络连接异常。

当发生异常时，后端应返回错误信息：

```json
{
  "status": "error",
  "message": "无法从视频流中读取图像，请检查 Atlas 推流是否正常。"
}
```

---

### 3.3 MiniCPM-V 4.0 图像理解设计

MiniCPM-V 4.0 是本系统的多模态分析核心。它能够同时接收图像和文本输入，并根据图像内容回答用户提出的问题。

由于 PC 使用 RTX 4060 8GB 显卡，显存有限，因此采用 MiniCPM-V 4.0 int4 量化版本，以降低显存占用，使模型能够在本地运行。

分析时，后端程序将从 RTSP 流中截取的当前帧保存为图像对象，并将其与用户输入的 prompt 一起送入模型。模型输出自然语言分析结果，例如：

```text
画面中可以看到一辆小车，位于图像中央区域。周围没有明显行人，整体环境较为空旷。
```

为了保证输出适合嵌入式界面显示，需要控制回答长度，避免过长。可在 prompt 前加入系统提示：

```text
你是一个嵌入式视觉分析助手。请基于当前画面回答用户问题，回答要简洁明确，控制在100字以内。
```

这样可以使返回结果更加稳定、清晰，便于 IMX6 端显示。

### 3.4 IMX6 Qt 交互界面设计

IMX6 端 Qt 程序主要完成用户交互，界面设计包括：

1. 标题区域：显示系统名称；
2. 文本输入框：用于输入用户分析提示；
3. 分析按钮：点击后发送请求；
4. 结果显示框：显示 PC 返回的分析结果；
5. 状态提示区：显示连接状态、分析中、分析完成或错误信息。

界面运行流程如下：

```text
用户输入提示词
      ↓
点击“分析”
      ↓
Qt 程序构造 JSON 请求
      ↓
通过 HTTP POST 发送给 PC
      ↓
接收 PC 返回结果
      ↓
在界面中显示分析文本
```

Qt 端可以使用 `QNetworkAccessManager` 实现 HTTP 请求。请求地址为：

```text
http://192.168.1.10:8000/analyze
```

发送 JSON 数据：

```json
{
  "prompt": "请分析当前画面中是否有行人。"
}
```

收到返回后解析 `result` 字段，并显示在界面文本框中。

## 5. 遇到的问题及解决方法

### 5.1 多设备网络连接问题

在系统设计初期，PC 只有一个网口，而系统中需要同时连接 Atlas 200 和 IMX6。若采用网线直连方式，只能连接其中一台设备，无法构成完整系统。

解决方法是使用交换机或路由器，将 Atlas 200、PC 和 IMX6 同时接入同一个局域网。这样三台设备可以通过 IP 地址互相访问，系统结构更加稳定，也更接近实际工程部署方式。

---

### 5.2 IP 地址配置问题

在局域网通信过程中，如果三台设备不在同一个网段，则会出现无法 ping 通、无法访问后端接口或无法推流的问题。

解决方法是统一规划 IP 地址，例如：

```text
PC:       192.168.1.10
Atlas:    192.168.1.11
IMX6:     192.168.1.12
```

并保证子网掩码一致，例如：

```text
255.255.255.0
```

调试时可先使用 `ping` 命令测试三台设备之间的连通性。

---

### 5.3 视频流读取问题

PC 端分析服务需要从 MediaMTX 中读取当前帧。如果 Atlas 没有正常推流，或者推流地址填写错误，PC 后端就无法获取图像。

解决方法是先使用 VLC 或 ffplay 独立测试 RTSP 视频流：

```text
rtsp://192.168.1.10:8554/live
```

只有在 VLC 能正常播放视频后，再启动后端分析服务，这样可以快速定位问题来源。

---

### 5.4 MiniCPM-V 显存占用问题

由于 PC 使用 RTX 4060 8GB 显卡，显存资源有限。如果直接运行较大的多模态模型，可能会出现显存不足的问题。

解决方法是采用 MiniCPM-V 4.0 int4 量化版本。int4 模型显存占用更低，更适合在 8GB 显存环境下运行。同时，可以限制输入图像分辨率和输出文本长度，以降低推理压力。

---

### 5.5 IMX6 端计算能力有限问题

IMX6 嵌入式实验箱计算能力有限，不适合运行目标检测模型或多模态大模型。如果将复杂 AI 推理任务放在 IMX6 上，会导致运行速度慢甚至无法运行。

解决方法是将 IMX6 设计为交互终端，只负责 Qt 界面、用户输入和结果显示。复杂计算任务全部放在 Atlas 200 和 PC 上完成，从而实现合理的任务划分。

---

## 6. 设计总结及设计心得

本次综合实验完成了一个基于 Atlas 200、PC 和 IMX6 的嵌入式多模态视觉分析系统。系统将边缘计算、视频流传输、大模型推理和嵌入式人机交互结合起来，实现了从视频采集、目标检测、视频推流、图像分析到结果显示的完整流程。

通过本次设计，我对计算系统中软硬件协同的思想有了更深入的理解。Atlas 200 适合承担边缘端视觉推理和视频处理任务，PC 端适合承担计算量较大的多模态大模型推理任务，而 IMX6 更适合作为轻量级交互终端。将不同任务合理分配到不同设备上，可以充分发挥各设备的特点，提高系统整体效率和稳定性。

在实现过程中，我也认识到网络通信在多设备系统中的重要性。由于系统包含三台设备，必须合理规划网络结构和 IP 地址。相比于简单的网线直连，使用交换机或路由器构建局域网更加稳定，也更符合实际工程系统的部署方式。

此外，本设计也让我熟悉了 MediaMTX 流媒体服务、RTSP 视频推流、HTTP 接口通信、Qt 网络请求以及多模态大模型调用等内容。系统不仅完成了基础的图像检测功能，还能够根据用户输入的自然语言提示进行语义分析，具有较好的交互性和扩展性。

总体来看，本系统实现了一个较完整的“边缘 AI + 多模态大模型 + 嵌入式交互”应用原型。后续可以继续扩展实时视频显示、多轮对话、异常事件检测、语音输入输出和 Web 可视化等功能，使系统更加完善。
